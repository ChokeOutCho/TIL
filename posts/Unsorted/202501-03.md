컴파일러와 인터프리터의 차이.

컴파일러는 전체 소스코드를 한 번에 번역하여 실행 파일을 생성하고, 독립적으로 실행할 수 있다. C, CPP, Rust, Go 등이 있다.
실행 속도가 빠르고 코드 최적화를 통해 성능을 향상시킬 수 있으며, 실행 파일만 배포하므로 소스 코드가 노출되지 않는다. 
그러나 소스 코드를 번역하는 데 시간이 걸리고 컴파일된 코드를 디버깅하기 어렵다.

인터프리터는 소스 코드를 한 줄씩 읽고 즉시 실행한다. Python, Ruby, JavaScript 등이 있다. 성능이 떨어지지만 컴파일을 생략하는 경우도 있고 포인터를 사용하지 않기 때문에 개발 속도가 빠르다. 프레임워크(.net, java 등)가 필요한 경우도 있다.

스택 메모리 상식

스택 영역의 최대 크기의 기본값은 1MB이다. 그래서 큰 영역 할당은 동적으로 해야한다. 이 크기는 작동 중 변경할 수 없고 컴파일 설정이나 스레드를 생성할 때 인자로 넣어서 변경해야 한다.

CPU는 현재 스택 프레임만 알고 있다. 그저 ebp esp만 기억하고 작동한다. 만약 사용자가 임의로 ebp와 esp값을 변경하면 cpu는 인지하지 못하고 그 값을 참조해 작동하게 되고, 리턴 시에 오류가 발생할 것이다.

실제 esp는 스택이 쌓일 수록 값이 낮아지게 된다. 이는 스택 메모리가 하향식으로 성장하기 때문이다.

스택 영역은 데이터 저장소일뿐, 실제 실행은 코드 영역의 명령어를 읽으면서 실행된다.


	
어셈블리 입장에서의 반복문(for).

반복문은 어셈블리 입장에선 비교와 점프로 이루어진 로직일 뿐이다. ecx를 사용하지 않는다. ecx, esi, edi 등은 어셈블리 전용 연산에서만 사용된다.

하나의 함수에 지역변수를 많이 할당하는 것이 일반적으로 성능에 미치는 영향과 그 이유.

지역 변수 할당은 esp 증감 연산만 수행한다. 그 이유는 스택 영역은 함수 호출시에 필요한 메모리 양만큼 할당되기 때문이다.

e.g. 400B할당 시 sub esp,400

^: XOR은 간단한 암호화에 주로 사용된다.

fastcall: 매개변수 중 일부는 레지스터에 전달되고 나머지는 스택에 푸시 된다. 이로 인해 호출 오버헤드를 줄일 수 있다. 일반적으로 피호출자가 함수를 정리한다.

const가 포인터 변수에 붙을 때 주의 사항
const가 포인터 변수에 붙을 때는 세가지 케이스가 있다.
    i) const int *p = &a;
    ii) int const *p = &a;
    iii) int * const p = &a;
이 중 i과 ii는 *연산자에 const가 붙은 케이스로 이 변수를 통해서 a에 접근해 값을 변경하는 것을 허용하지 않지만 p가 가리키는 변수를 변경할 수는 있다.
iii는 둘과 반대로 p가 가리키는 변수를 변경할 수는 없지만 *p를 통해 a의 값을 변경할 수는 있다.

int의 위치는 무시하고 const와 *의 위치만 신경쓰는 편이 헷갈리지 않는다.

테스트 중에는 변인통제와 문제 상황 재연을 위해 씨드를 고정시키곤 한다.

구조체 멤버 변수 패딩은 구조체에서만 적용되는가?

구조체 멤버변수 패딩은 흔히 구조체의 가장 큰 변수 크기에 align된다는 소문이 있다. 하지만 실제로는 자신의 데이터 타입에 맞는 정렬을 따르며 이는 딱히 멤버변수에만 적용되는 사항이 아니다. 모든 변수들은 자신의 타입에 맞는 정렬을 따르고 있다. 예를 들면 short는 2로 떨어지는, int는 4로 떨어지는 메모리 경계에서 할당된다.


따라서
+-----------+---------------------------------+
|    char a      |
+-----------+—------------------------------- +
|    short b     | 이 부분만 보고서는 패딩을 알 수 없다.|
|    int c        |					       |
+-----------+ —-------------------------------+
만약 a가 존재 하지 않는다면 둘 사이의 패딩은 2가 되지만 a가 존재하면 패딩이 사라지기 때문이다.

변수들이 자신의 정렬 경계에 서야 cpu가 한번에 접근할 수 있기 때문에 빠르다고는 하지만 사실 성능적으로는 티가 크게 나지 않는 편이라고 한다. 

통신에서의 사용:
통신 시 패킷으로 사용할 땐 패딩을 없애기도 한다. #pragma pack(n)이나 프로젝트 설정/코드 생성/구조체 멤버 맞춤/ 을 통해서 변수를 n으로 정렬하는 전처리를 할 수 있다. 그러나 모든 변수를 n으로 정렬하는 것이 아닌 n보다 큰 변수를 정렬하라는 의미이다. 따라서 32bit 시스템에서 n을 4로 설정하면 아무것도 변하지 않을 것이다. #pragma pack을 사용했다면 n에 0을 삽입하여 다시 돌려놓지 않으면 이후 모든 변수가 n으로 정렬되기 때문에 반드시 돌려놓아야 한다. 이 방법은 일시적으로 속도를 살짝 포기하고 네트워크 트래픽을 챙길 때 유용하다.

캐시와의 관련성:
64비트 시스템에 와서는 cpu가 캐시 라인 단위(64byte)로 읽어가기 때문에 읽기 쓰기(load와 store 혹은 mov)에 대해서 원자 단위 연산을 보장한다. 이 때 원자적은 내가 바라본 값을 온전히 가져온다는, 즉 가져오는 동안 다른 프로세스에서 값을 변경할 일이 없음을 보장한다는 의미이다. 캐시는 64byte 덩어리로 긁어가기 때문에 이 범위에만 들어오면 원자적이다. 하지만 여전히 cpu는 word 단위로 동작하기 때문에 align하지 않으면 좋지 않다.

멤버 변수의 순서를 결정할 때:
구조체 접근은 첫 멤버의 포인터에서 +n을 통해 접근한다. 즉 첫 멤버는 증가 연산을 사용하지 않는다. 가장 자주 사용하는 변수를 첫 멤버로 결정하고 그 이후는 캐시 등 여러 요소를 고려해야 하지만 대부분의 상황에서 메모리를 절약하기 위해 멤버의 순서를 바꾸지는 않는다.

구조체는 가장 큰 멤버변수의 크기에 맞춰 정렬된다:
구조체 멤버변수들은 자신의 크기를 기준으로 정렬되지만 구조체의 시작은 가장큰 멤버 변수의 크기를 기준으로 정렬되어야 한다. 그렇다면 구조체 자체에 패딩이 생겨 메모리가 낭비된다고 생각할 수도 있지만 구조체를 배열로 선언한다면 패딩이 있는 편이 접근할 때 더 효율적일 것이다.

전역변수는 위치가 고정된다?
대부분의 코드는 기준 주소(base address)기준으로 빌드된다. 함수나 전역변수에 접근하거나 점프하는 명령어는 절대 주소와 상대 주소를 대상으로 사용 가능하며, 절대 주소를 사용한 경우에는 명령어 바이트에 명시된다. 그러나 절대 주소도 기준 주소에 오프셋(base + 주소)으로 접근하기 때문에 실행 파일에 주소가 그대로 명시되는 것은 아니다.

dll은 주소가 겹칠 가능성이 있기 때문에 주소 인덱스를 가지고 있으며, 로드될 때 주소가 다시 계산되기 때문에 이 또한 실행 파일에 명시되지는 않는다.

어셈에서 절대 주소로 jmp했는데 실제 코드 바이트에서 찾을 수 없는 경우가 있는데, 이는 디스 어셈블리가 상대주소를 미리 계산 해 놓은 것이다.

전역 변수 라도 상대 주소로 나오는 경우가 있다.

alignas

캐시와 메모리 경계가 연관되면 더 큰 경계를 원할 때가 올 것이다. 이 때 사용되는 키워드가 alignas(n)인데 이는 변수를 n바이트 경계 주소에 세우겠다는 의미이다.
$$
char a;
alignas(16) int b;
short c;
alignas(32) int d;
$$



두 가지 고려할 사항:

규격화된 힙에서 정상 작동 할 것인가?

기본적인 동적할당 malloc은 정렬을 지원하지 않는다. 힙은 시스템 마다 8바이트 혹은 16바이트로 규격화 되어 있다. 즉 재수가 좋아 8바이트 16바이트로 떨어지는 경우를 제외한다면 정렬은 불가능하다. 메모리 정렬이 필요한 동적할당은 _aligned_malloc을 사용한다.

시스템크기 단위로 움직이는 스택에서 정상 작동 할 것인가?

과거 alignas가 cpp표준이 되기 전에는 정렬에 Visual C에서 만든 declspec(align(#))가 사용되었는데, malloc과 같은 이유로 align이 정상 작동하지 않았으나 alignas가 cpp표준이 되면서 가능하게 되었다.

가상 메모리와 힙

가상 메모리 목적: 프로세스 간 분리, 효율적인 메모리 사용

가상 메모리는 운영체제가 제공하는 것이 아닌 cpu의 설계이다. 과거의 메모리 시스템은 다른 프로세스의 메모리에 접근이 쉽기 때문에 위험하다. 가상 메모리는 물리 메모리의 4kb 프레임이 가상 메모리 테이블을 통해 4kb페이지와 매핑되는 메모리 관리 방법이다.

virtualAlloc: 우리가 접근 가능한 가장 로우 레벨 메모리 할당이다. 이는 페이지 단위로 할당하기 때문에 복잡하다. 그래서 운영체제에서 heap을 제공한다. 우리가 힙 메모리라고 부르는 것의 실체는 운영체제 코드일 뿐이었던 것이다. 힙은 복잡한 페이지 단위의 할당을 대신 수행해주는 주체이다. 
즉 동적 할당시에는 new -> malloc -> heapAlloc -> virtualAlloc 순으로 호출된다는 의미이다.

그러니 힙, 스택과 메모리를 혼동해서는 안된다. 이들은 언어적인 관점일 뿐이고 실제론 이러한 것에 속하지 않는 PCB나 TCB등이 존재한다. 스택도 최대는 1mb이지만 내부적으로는 4kb단위로 늘어난다.


페이지 커밋과 메모리 접근시 에러 발생의 연관성

커밋이란 페이지와 프레임이 가상 메모리 테이블에서 매핑된 상태를 의미한다. 메모리 접근 시에 에러가 발생하려면 접근 위치가 쓰기가 금지되어 있거나, 프리(??로 표기 된 메모리)거나, 커밋이 안되어 있어야 한다.
$$
char * p = malloc(20);
free(p);
*p = 10;
$$
그러나 위 코드는 p에 접근이 가능하다. free()가 되었지만 힙이 할당 해제된 것이지 페이지가 커밋 해제된 것은 아니기 때문이다. 해당 접근에서 오류가 발생하려면 커밋이 되지 않아야 한다. 때문에 위 코드에서 오류를 발생 시키는 경우는 없다.

힙과 페이지

힙 메모리 내에서 할당한 블록들의 사이에는 힙 정보(메타데이터)가 들어간다. 이는 메모리 뷰에서도 확인할 수 있다. 그리고 힙은 처음 동적 할당 받았던 자리가 아닌 위치에서 free()하면 heapFailure을 일으키며 수행하지 않는다.

+-----+----+-----+------+---+
|*p      |meta|*q     |meta   | …  |
+-----+----+-----+------+---+ <- 4kb페이지 내에 할당 된 동적 메모리 블럭

페이지가 해제되려면 요소들이 모두 제거되어야 하는데, 페이지 내의 메모리 블럭들이 모두 사라진다고 해서 곧바로 페이지가 해제되는 것은 아니다. 내부적으로 페이지 사용 빈도를 측정해 몇 개의 페이지가 빈 후에야 해제되는 동작이 있을 것이기 때문이다.

만약 페이지가 해제되는 순간을 보고싶다면 엄청 큰 동적 할당은 힙이 관리하지 않는 것을 이용해볼 수 있다.
malloc(1024*1024*5); 처럼 엄청 큰 블록을 할당하게 되면 메모리 아래 쪽 어딘가로 홀로 존재하게 될텐데(주변 메모리가 ??임), 이를 할당하고 해제하면 메모리 뷰가 ??로 날아가는 모습을 볼 수 있고, 해당 동작이 페이지 해제이다.

메모리에서 페이지의 시작주소는 4kb의 배수가 된다. 즉 4096의 배수라면 페이지의 시작 주소일 수 있다. 12비트 최대값은 4095이기 때문에 만약 주소가 0x007FF000이라면 페이지의 시작 주소일 가능성이 높다는 것이다.

시스템에서 가상 메모리 모니터링

고급 시스템 설정에 페이징 파일 크기를 설정할 수 있다. 페이지 아웃될 때 덜 사용된 메모리 페이지를 디스크의 페이징 파일에 옮긴다. 페이지 아웃된(물리 메모리에 올라와 있지 않은) 데이터에 접근한다고 에러가 발생하는 것은 아니지만 페이징 파일에서 페이지 인 하는 속도가 느리기 때문에 성능 저하가 발생한다.

스와핑은 물리 메모리가 부족할 때 일어나기도 하지만 딱히 부족하지 않아도 오래 사용되지 않는 페이지 또한 페이지 아웃 시킨다. 페이징 병목을 해결하기 위해 램이 충분하게 준비되어 있다면 페이징 기능을 비활성화 시키거나 물리적인 디스크를 더 사용해서 페이징 전용으로 사용하는 경우도 있다.

작업 관리자에서 커밋 크기를 활성화하고 변화량을 모니터링하여 메모리 누수 여부를 판단하는데 사용할 수 있다. 이 때 개인 집합은 내 프로세스가 혼자 사용하는 메모리를 의미한다. 데이터, 힙, 스택 등의 private 메모리가 여기에 포함된다.
$$
char *p = malloc(1024(kb)*1024(mb)*1024(gb));
for(int i = 0; i<p_length)
{
	*p = 0;
	p += 1024;
}
$$
이 코드를 실행하고 커밋 크기를 살펴보면 커밋크기가 바로 1기가 할당되고 개인 집합 크기가 슬금슬금 올라간다. 이 경우에는 커밋 크기는 변하지 않고 있기 때문에 누수가 발생하고 있다고 단정짓기 어렵다. 이는 데이터에 접근했을 때 물리 메모리에 매핑하는 메모리에 대한 지연 처리이다. 커밋 크기는 당연하게도 페이지 미만의 크기는 추적할 수 없는데, 실제로 작은 크기의 누수가 있더라도 커밋의 변화가 없다면 큰 메모리 누수는 없다고 판단할 수 있다. 그러나 디버그 모드에서는 디버그 전용의 malloc이 값을 밀 때(cdcdcd..) 즉시 접근하기 때문에 활성 개인 집합이 즉시 거대하게 잡힌다.

페이지 폴트 항목

CPU에 대한 에러는 운영체제가 먼저 개입한다. 이는 하드웨어 인터럽트 또는 TRAP이라고 한다. 페이지 폴트를 CPU가 발생시키면 운영체제가 핸들링한다.

페이지 엔트리에서 유효비트가 0이면 전부 페이지 폴트이다. CPU 입장에서는 페이지 접근 에러이기 때문이다. 

페이지 아웃 시에는 해당 페이지의 유효 비트를 0으로 설정한다. 이러면 페이지가 무효되어 없는 상태와 같아진다. 이후 페이지의 정보를 페이지 아웃한 디스크의 정보로 채운다. 이 페이지에 CPU가 접근하면 페이지 접근 에러가 발생하고 운영체제가 페이지 아웃임을 판단하면 디스크 정보를 보고 페이지를 가져와 커밋할 것이다.

MMF에서 볼 수 있는 파일 이미지를 실행 됐을 때 메모리에 지연 로드하는 경우도 페이지 폴트이다. 메모리에 매핑되었지만 아직 물리 메모리에 로드되지 않은 페이지에 접근하면 페이지 폴트가 발생한다. 이는 아웃된 페이지를 참조하는 경우와 유사하다. 운영체제는 파일에서 해당 페이지를 물리 메모리에 적재하고 정상적으로 실행하고 이 동작은 유저가 알 수 없다.

권한 위반 페이지폴트:
페이지의 보안 특성에 의해 발생하는 예외는 권한 위반 페이지 폴트이다. 
즉, Copy on write의 동작도 권한 위반 페이지 폴트이다. 

권한 위반 - Copy on write:
Copy on write 보안 특성은 여러 프로세스가 동일한 메모리를 읽기 속성으로 참조할 수 있게하는데에 필요하다. exe나 dll같은 파일 이미지는 여러 프로세스가 같은 영역을 공유할 수 있는데, 특정 프로세스가 공유된 영역(페이지)을 수정하려는 동작을 감지하면 변경하려는 페이지를 해당 프로세스에게 독립적인 쓰기 속성 페이지로 복사한 후 매핑을 그 페이지로 변경한다.

한 프레임의 경과시간을 매번 시간 함수로 가져와 계산하는 것이 아닌 상수로 증가시켜서 오차를 보정해주는 방법이 좋겠다. 한 프레임을 실행하는데 실제 시간이 25ms증가했다면 5ms는 다음 프레임의 경과시간으로 넘기고, 17ms가 걸렸다면 3ms만큼 대기한다. 빠르든 느리든 경과시간은 상수로 증가시키게 된다. 경과시간은 현실 시간을 따라잡는 가상의 시간이 될 것이다.

위의 방식으로 구현했을 때 경과시간을 다음 프레임으로 미루다 한 프레임의 기대시간을 넘기는 순간의 처리 방식에 따라 렌더링 스킵과 FixedUpdate로 나뉘게 된다.

필요없는 것을 포기하는 방법(렌더링 스킵):
클라의 경우에는 렌더링을 버리는 형태가 된다. 렌더링은 일반적으로 로직보다 훨씬 많은 cpu리소스를 잡아먹는다. 한 프레임에서 렌더링을 하지 않으면서 로직을 따라잡으며, 경과시간을 상수로 누적시키면서 실제 시간을 따라잡는 형태로 구현한다. 만약 경과시간과 실제시간의 차가 50ms라면 렌더링이 두 번 스킵되지만 로직은 기대한 만큼 두 번 완료되어 화면이 뚝 뚝 끊기는 것처럼 보인다.

부족한 로직을 더 돌리는 방법(FixedUpdate):
사실 렌더링 스킵과 굉장히 유사하다. (실제시간 - 경과시간)이 50ms라면 로직을 빠르게 두 번 더 돌리고 경과 시간을 20ms*2 누적시킨다. 이러면 차이는 10ms가 남고 정상 속도로 복구될 것이다.

두 가지 방법은 그저 구현의 차이이고 로직의 호출 횟수를 보장한다는 목적은 동일하다. 이 방법으로 프레임을 관리한다면 사실 시간당 프레임이 고정된 루프문에서는 시간에 의존하는 로직과 프레임에 의존하는 로직이 같은게 된다.

소멸자가 없는 클래스 대상의 delete도 마찬가지로 free이다. 소멸자가 존재하는 객체의 delete는 스칼라와 벡터버전으로 분기된다. 이유는 아래에서 설명한다.
new를 호출할 때 객체의 소멸자가 정의되었으면 단독 객체인 경우와 객체 배열일 경우의 동작이 달라진다. 단독 객체는 기존과 동일하지만 객체 배열일 경우에는 실제 할당한 주소에서 4증가된 주소값이 반환된다. 이는 소멸자 호출의 횟수를 저장해야하기 때문이다.

만약 소멸자를 지닌 12byte의 객체 10개를 배열로 생성한다면 124byte를 할당 받는다는 것이다. 할당한 영역의 최상단에는 4바이트의 정수를 저장하는데, 이는 배열 원소의 개수인 10일 것이다. 이 영역의 크기는 x86 x86-64 모두 동일하다. 4byte로 표현불가능한 정수보다 많은 원소 개수를 가질 일은 거의 없을 것이다. 4byte 영역 이후에는 120byte로 원소를 표현한다. 이후 처음 할당 받은 주소에서 4byte 증가한 주소(배열의 시작 주소)로 반환받게 된다.

delete는 객체의 소멸을 보장할 의무가 있다. 때문에 소멸자를 완벽히 호출한 후 해제 해야한다. 그러나 delete는 인자는 포인터로, 타입과 주소만 알고 있다. 하지만 객체 배열은 원소의 수만큼 소멸자를 호출해야 하기 때문에 원소의 수를 알 필요가 있다. 이 정보는 생성할 때 추가로 할당받은 4바이트 영역에 적재해두었다. 

소멸자가 정의된 객체배열은 vector버전 delete가 호출되는데 할당받은 첫 영역의 4바이트에 담긴 정수만큼 소멸자를 호출하고 메모리를 해제할 것이다. 
메모리를 해제할 때에는 당연히 처음 할당했던 주소를 인자로 넣어야 한다. vector버전의 delete는 배열의 시작주소를 인자로 받아 4를 감소시킨 할당 시작주소로 변경하여 해제를 시도한다. 이렇게 하면 원소들의 소멸자를 모두 해제시킨 후 메모리를 해제할 수 있다.

만약 이 4바이트 정수가 오염되었다면 오염된 데이터만큼 소멸자를 호출하다가 의도하지 않은 동작이 발생할 것이다. 만약 소멸자에서 터졌다면 이 정수 영역을 확인해보는게 좋겠다. 이는 delete를 스칼라와 벡터 서로 다른 버전으로 호출했을 때에도 동일하다. 
만약 소멸자가 있는 스칼라 객체를 벡터 버전 delete하면 상위 4바이트 만큼 소멸자를 호출하는 이상한 동작을 보일 것이다. 혹은 정상 동작으로 보일지라도 free에서 예외를 뱉어낼 것이다. 해제를 요청한 주소가 할당받은 주소와 일치하지 않기 때문이다.

단일 객체를 벡터버전으로 delete했을 때엔 수많은 호출을 반복하면서 this를 참조하며 데이터를 오염시키다가 유효하지 않은 메모리에 접근했을 때 터진다. 그런데 최근에는 소멸자 호출 반복을 뒤에서 부터 하도록 구조가 변경되어 처음 호출에서 터진다고 한다.

delete 연산자의 소소한 기능으로 해제한 공간에 0x8123을 넣는 동작이 있다. 이는 잘 해제 되었음을 의미한다. 소멸자가 없다면 그냥 mov로 0x8123을 넣는다. 이 또한 보안 검사 옵션이다.

캐시

인텔CPU는 코어마다 L1과 L2가 붙고 L3는 공유된다. 캐시는 단순히 표현하면 메모리를 사본 뜬 것이며, 캐시 히트시의 성능을 상수로 비교할 수는 없지만 메모리 접근보다 40배 정도 빠르다고 알려져 있다. 캐시는 이름부터 은닉처이고 프로그래머에게 투명하지만 우리는 캐시 히트를 유도할 필요가 있다.

캐시라인은 현재는 64byte이고 a변수를 읽고 싶을때에는 당연히 a가 포함된 캐시 라인을 긁어간다. 이 때 캐시 라인의 시작주소는 a변수를 64byte로 내린 주소이다.

volatile이 캐시를 거치지 않는다는 소문이 퍼져있지만 프로그래머가 사용한 데이터가 CPU로 전달되려면 반드시 캐시를 거쳐야 한다. volatile은 그저 최적화 컴파일을 막는 키워드일 뿐이다. 만약 a라는 변수가 레지스터에 있다면 최적화되어 바로 레지스터에서 값을 가져다 쓰는 경우에는 캐시를 통하지 않고 동작하기 때문에 소문이 와전된 것으로 보인다.

DMA는 외부장치 IO에 쓰이는 장치인데, 이 장치는 CPU를 거치지 않고 메모리에 직접 데이터를 쓴다. 다만 이러한 경우에는 CPU가 인식하지 못하기 때문에 이 장치가 사용한 메모리는 보호특성 NO_CACHE가 적용되어 캐시로 긁어가지 않는다.

캐시를 사상(매핑)하는 방법에는 크게 직접 사상과 연관 사상이 있다. 한글 용어는 서적마다 조금씩 차이가 있다. 하지만 직접 사상 방식은 메모리 블록이 정해진 캐시 라인에만 사상이 가능하고, 완전 연관 사상 방식은 메모리 블록이 어떤 캐시블록에도 사상 가능하며 검색 시 모든 태그를 순회해야 한다. 현재 인텔은 둘을 절충한 집합 연관 사상 방식을 사용하고 있다. 직접 사상처럼 정해진 장소에만 사상될 수 있으나 완전 연관 사상처럼 정해진 장소 내에서는 어느 캐시 블록에든 사상될 수 있다. 정해진 장소(인덱스)에 2개 이상의 블록으로 구성된 집합으로 구성될 수 있다. 2-way 집합 연관 사상이라면 2개의 캐시 블록 중 어디에든 메모리 블록을 사상할 수 있다.

캐시 히트율은 ( 1 - 캐시 히트 수 / 총 메모리 접근 수)로 표현할 수 있고 캐시 히트를 높이려면 인덱스를 가능한 넓게 펼쳐서 사용해야 한다. 일반적으로 캐시 히트율을 높이는 방법에는 시간 지역성과 공간 지역성이 있다. 시간 지역성은 단순히 최근에 접근한 메모리에 다시 접근하는 경향이 있다는 것을 의미한다.

캐시 미스는 요구한 캐시라인과 기존 캐시라인의 인덱스가 같은 경우이다. 이 경우에는 기존의 캐시라인을 밀어내고 새로 요구한 캐시라인이 적재될 것이다. 밀어내는 기준에는 선입선출 혹은 최소최근이 사용될 수 있다.

!공간 지역성! :
상기한 내용 중 캐시 히트율을 높이려면 인덱스를 최대한 넓게 펼쳐서 사용할 필요가 있다고 언급했다. 만약 같은 로직인데 속도의 차이가 있다면 인덱스가 겹쳐 캐시 미스가 자주 발생한 것은 아닌지 변수의 주소를 계산해서 검증해야 할 것이다. 현재 이 문서를 작성하는 컴퓨터의 CPU스펙은 다음과 같다.
Intel Pentium Gold G5420, 코어 2개, L1 캐시 128kb (L1.Data 2*32 8-way, L1.Inst 2*32 8-way)

이 CPU의 L1캐시의 인덱스 비트 수는 32bit 시스템에서 다음과 같이 구할 수 있다.
$$
128kb * 1024 = 131,072 	// 이 저장소의 크기는 131,072바이트 이다.

131,072 / 2 = 65,536 	// L1은 코어마다 하나씩 붙는다. 이 CPU는
// 더블코어이다.

65,536 / 2 = 32,768		// 데이터와 명령어 캐시가 절반씩 사용된다.

32,768 / 64 / 8 = 64		// 하나의 캐시 라인은 64바이트 이고, 하나의
// 인덱스에는 8개의 캐시라인이 들어갈 수 있다. 즉,
// 인덱스의 수는 64이다.
$$
이 CPU의 L1캐시의 인덱스 64개를 표현하는 데 비트가 6개 필요하다. 그리고 캐시라인의 오프셋 64바이트를 표현하는 데 비트가 6개 필요하다. 따라서 인덱스 6비트 + 캐시라인 오프셋 6비트를 제외한 나머지 상위 20비트는 태그가 될 것이다.

사실 4kb 페이지를 사용하는 시스템은 인덱스와 캐시라인 오프셋을 더한 크기가 12bit인 사상 방식을 사용해야 한다. 캐시가 사용하는 주소는 결국 물리 주소가 될텐데, 하위 12bit로 표현하면 페이지의 크기와 일치하고, 이는 선형 주소 변환의 대상이 아니기 때문에 가상 주소를 그대로 사용할 수 있는 것이다.

공간 지역성의 진짜 의미는 12비트 == 4kb == 페이지 1개의 크기이고, 인덱스 64개의 way하나씩을 사용하면 페이지 1개를 펼쳐 적재할 수 있다는 것이다. 즉 이 L1캐시는 way를 8개 사용하기 때문에 페이지 8개를 펼쳐서 적재할 수 있다.

MESI프로토콜

캐시 라인마다 메타데이터로 4가지 상태가 존재하며 이 상태를 유지 및 갱신하는 규칙을 MESI프로토콜이라고 한다.

캐시라인의 64바이트로 캐시라인 하나의 주소는 인덱스로 6비트 오프셋으로 6비트 외의 비트를 태그로 사용한다고 했다. 태그는 물리주소로 변환되고 이 주소를 캐싱하는게 TLB이다.

Modify: 캐시라인이 수정되어 다른 저장소들과 데이터가 다르다.
Exclusive: 캐시라인이 단일 캐시에 유일하게 존재하고 데이터가 RAM과 동일하다.
Shared: 캐시라인이 다른 캐시에도 존재하고 데이터가 RAM과 동일하다.
Invalid: 캐시라인이 무효화 되었다.

캐시와 메모리의 동기화는 write through와 write back 동작이 있다. 전자는 즉시 반영, 후자는 퇴출될 때 반영이다. 전자의 경우는 코드 캐시(instruction cache)에서 사용된다. 코드는 바뀔 일이 거의 없기 때문일까?

캐시 무효화는 한 캐시에서 Modify상태로 변경하면서 다른 캐시의 같은 캐시라인의 invalid처리를 broadcast 하면서 발생한다.

A스레드는 변수 a에 대해서 읽기만 수행하고 B스레드는 같은 캐시라인의 변수 b에 대해서 쓰기를 실행한다고 가정해보겠다. 이 캐시라인은 B스레드가 수정했기 때문에 A스레드가 들어간 코어에서 무효되어있을 가능성이 높다. 때문에 A스레드를 처리한 코어의 L1캐시에서 미스가 발생하여 B스레드를 처리한 코어의 캐시에서 Modify로 변경한 캐시라인을 L3에서 가져와야 할 것이다. 이 때 상태 규칙을 지켜야 하기 때문에 Shared로 변경되기 전에 a가 포함된 캐시라인이 메모리에 반영되야 할 것이다. 메모리에 변경된 캐시라인을 반영한 그 한순간에 잠시 Exclusive로 변했다가 B스레드가 캐시라인을 가져가면서 Shared상태로 변경할 것으로 상상..하고있다.

캐시 프로토콜 내에 상태 규칙을 지키기 위한 동기화 수단이 있을 것으로 보인다. 실제 위 코드는 읽기와 쓰기 모두 실행속도가 현저히 떨어졌었는데 단순히 위 이론을 따르면 쓰기를 진행한 코어에서는 캐시 미스가 발생할 이유가 전혀 없기 때문이다. 만약 쓰기를 진행한 코어가 느려진 이유가 메모리에 접근하기 때문이라면 상식적으로 L3캐시의 접근 속도보다 메모리로의 접근 속도가 훨씬 더 느리기 때문에 매우매우매우 느려져야한다.

A스레드(읽기)를 실행한 코어는 A코어, 반대(쓰기)는 B코어라고 부르겠다. 위의 현상을 설명하기 위해서는 B코어가 A코어의 동작을 기다렸다고 의심해볼 수 있다.

i) 	A코어에서 미스가 발생해 B코어로부터 캐시라인을 요청했다.
ii) 	B코어는 MESI프로토콜을 지키기위해 메모리에 쓰는 작업을 수행했다.
iii) 	이후 A코어가 B코어로부터 캐시라인을 공유받았다.

개인적으로 캐시 프로토콜은 고민해도 의미가 딱히 없다고 생각한다. 프로그래머 입장에서 취할 수 있는 대처라곤 같은 캐시라인에 놓지 않거나 인덱스가 겹치지 않도록 하는 것 밖에 없기 때문이다. 그리고 한 캐시라인의 동기화문제는 코드 선에서 배제시킬 것이기 때문이다. 게다가 비순차적 처리와 합쳐진다면 더더욱 예측할 수 없게 될것이다.

캐시 히트 유도 방법

캐시 히트를 100%유도하는 방법은 없다. 프로그래머는 당장 내 코드가 어떤 코어에서 실행될지도 알 수 없기 때문이다(우선도는 지정할 수 있지만). 하지만 캐시 미스의 예측은 100% 가능하다. 방법은 매우 쉬운데, 그저 인덱스가 겹치는 캐시라인이 way보다 많은지만 확인하면 된다.

캐시히트를 유도하기 위해서 64byte align 한다는 이야기도 있는데, 이는 캐시를 대놓고 낭비하겠다는 소리로 들린다. 만약 정말 자주 수행되는 로직이고 패딩도 얼마 안들어간다고 판단되면 고려해볼 수는 있겠지만 다른 로직이 느려질 수 있다는 사실을 인지하고있어야 한다.

위의 현상처럼 읽기만하는 데이터와 쓰기를 하는 데이터가 같은 라인에 적재돼있고, 이가 정말 자주 호출된다면 항상 캐시미스나서 로직을 느리게 만들 가능성이 다분하지만 이 때문에 이 둘을 분리한다는건 정말 말도 안된다. 자연스럽게 64바이트 경계에 세울 수 있으면서 다른 로직에 영향이 적을거라는 확신이 있는 상황 같은 까다로운 조건을 만족시킬 수 없다면 멍청한 최적화는 피해야 한다.

막연하게 어떤 데이터의 히트를 유도하는 것이 아니라 흐름을 보아야 한다. 로직의 흐름을 같은 캐시라인에 담아낼 수 있거나, 전혀 다른 흐름인데 같은 캐시라인에 묶여있다면 정말 조금은 고려해볼 법도 하다. 그러나 이 로직에 투자한다면 다른 로직이 느려질 가능성은 감안해야 한다.

당연히 이러한 사안을 구현 중에 고려하는건 불가능하다. 다만 로직이 지연되는 원인 중 하나로 나타날 가능성이 있고 이를 적용하는건 그 순간의 상황을 고려해보아야 한다. 최근의 시스템에서 로직이 느려지는 이유는 거의 메모리나 캐시가 원인이기 때문이다. 함수의 위치를 다르게 하니 속도가 달라졌다는 사례도 있다. 캐시 히트를 높이는 다양한 상황을 재현하고 고민해봐야겠다.

가상함수
가상함수는 부모 포인터로 실제(자식)의 멤버함수를 호출하게 하는 것이다. 즉 포인터 참조가 아니라면 가상함수가 작동하지 않는다.

try-catch는 seh의 랩핑이다. 할 수 있는 능력에서 차이가 있다. SEH는 시스템에서 발생하는 모든 종류의 예외를 핸들링할 수 있게 된다.

try-catch를 쓰는 이유는 stack unwinding때문이다. throw를 던졌는데 아무도 핸들링하지않으면 책임이 스택프레임의 밖으로 날아간다. main에서조차 핸들링 하지 않았다면 main도 벗어나서 프로그램을 종료시킨다. 
사실 stack unwinding은 스택을 하나씩 되감기 하는게 아니라 점프랑 똑같다. throw의 타입이 다르다면 자신의 핸들링이 아니기 때문에 모두 한번에 건너뛰어버려서 디버깅을 힘들게 만든다. 호출 스택을 한번에 닫아버리기 때문이다. 만약 try-catch를 사용한다면 전용 객체나 구조체를 만들어서 이것들을 던지게 된다. 

그런데 지역객체가 존재한다면 cpp는 객체의 소멸을 보장해야 하기 때문에 이들의 소멸자를 모두 호출한다. 이 동작을 위해 지역 객체가 try-catch문 내에서 사용됐고 해당 흐름이 예외를 발생시킬 가능성이 조금이라도 있다면 객체를 생성할 때 예외처리기를 등록한다. 이 때 어셈블리가 굉장히 더러워진다. 때문에 등록하는 부분에서 조금 느려지지만 다른 코드에는 딱히 영향이 없다. throw가 call throw로 바뀌는 정도이다.
등록된 예외처리는 스레드마다 리스트로 할당되고, throw되서 스택을 풀때 이를 참조해서 소멸자를 호출시킨다. noexcept는 해당 함수가 예외를 방출하지 않음을 명시하는 키워드이다. 만약 예외처리기 등록이 꺼려지는 상황이라면 이 키워드를 아주아주 꼼꼼히 써서 회피해야한다.

예외처리기는 어떠한 객체가 아니라 스택의 주소가 박혀있는 코드 덩어리 형태로 컴파일러가 생성하는 것으로 보인다. 정확히 공개되어있지는 않다. 예외를 처리할 때엔 마지막에 등록된 예외처리기부터 점프뛰어가며 소멸자를 호출한다.

네트워크도 IO이기 때문에 비페이징 풀이다. 송수신량이 쌓이면 시스템이 죽게 된다. 비페이징 풀이 최대치에 다다르면 당연히 로깅도 하지못한다. 요즘 서버는 이에 대한 대비를 하지 않아서 여길 공격하면 쉽게 죽일 수 있다.

윈도우의 커널모드와 cpu가 빨라진 이유
윈도우는 두 가지 모드가 있다. 하드웨어에 접근 가능한 ring 0와 사용자 모드인 ring 3이다. 우리는 ring 0를 윈도우의 커널이라고 부르곤 한다.

요즘은 프로세스가 아니라 스레드 기반이기 때문에 문맥교환이라고 하면 cpu에서 스레드가 바뀌는 것이다. 커널 모드를 전환할 때에도 문맥교환에 준하는 동작이 이루어지는데, 커널 모드로 전환되면 커널 스택으로 바뀌고 레지스터도 모두 교체된다.

요새 cpu가 빨라진 이유는 비순차적 처리 덕분이다.
비순차적 처리는 결국 명령들을 미리 계산해두고 이를 순차적으로 실행시키는 처리 방식인데, 인터럽트같은 이유로 문맥교환이 일어나면 미리 계산한 결과를 모두 버려야 한다. 때문에 커널모드를 유발하는 동작은 지양하는편이 좋은 성능을 가져올 수 있다.

인자로 커널 오브젝트를 받는 api들은 전부는 아니지만 높은 확률로 커널모드를 유발한다. 혹은 Sleep()과 같이 받지 않아도 커널모드를 100%유발하는 함수도 있다. mutex나 세마포 또한 커널 동기화이기 때문에 커널모드로 진입한다.

소켓도 커널오브젝트이기 때문에 소켓IO도 커널모드를 유발하지만, 다른 방법이 없기 때문에 사용해야한다. 다만 모드 전환을 최소화 하기 위해 버퍼를 사용하여 한번에 처리하여 반응성과 성능사이에서 적절한 교환을하고 있다.

여유 공간이 여러 조각으로 나뉘는 현상. 공간을 다양한 크기로 할당하고 해제할 때 일어나며, 할당 알고리즘이 약화된다. 
‘외부’라는 용어는 사용할 수 없는 기억 장소가 할당된 영역 밖에 있다는 것을 뜻한다.
비록 남은 공간을 사용할 수 있어도 조각이 너무 작게 나 있어서 성능이 악화되겠다.

“남은 전체 메모리 공간의 합은 충분하지만, '연속된' 공간이 없어서 큰 메모리를 할당하지 못하는 상태”

내부 단편화:
기억 장치가 의도된 바 없이 할당될 때 일어난다. 
‘내부’라는 용어는 공간이 할당된 영역 안에 있지만 쓰이지 않는 것을 의미한다.
페이징에서 발생할 수 있다.
쌩 가상 메모리는 1바이트를 사용하더라도 4kb가 커밋돼야한다. 이경우 나머지 4095바이트의 공간은 낭비된다.
