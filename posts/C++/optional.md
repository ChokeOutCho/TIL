
# std::optional
C++17부터 도입된 값이 있을 수도, 없을 수도 있는 객체를 표현하는 Wrapper클래스이다.
기존 C++에서는 값이 없음을 -1, nullptr, EOF등의 특수 값을 사용했지만 std::optional은 이를 타입 시스템 차원에서 해결한다.

## 1. 핵심 개념
std::optional<T>는 내부에 T타입 객체와 그 객체가 현재 존재하는지 나타내는 bool플래그를 함께 가진다.
* 값이 있는 경우: T 타입의 데이터가 있음
* 값이 없는 경우: std::nullopt 상태를 가짐.
```
// optional의 대략적인 메모리 형태
template <typename T>
class optional {
    bool m_hasValue; // 값이 있는지 확인하는 플래그
    union {
        char m_dummy; // 정렬을 위한 공간
        T m_value;    // 실제 데이터
    };
};
```

## 2. 효능
* 2.1. 의도 명확: 함수의 반환형에서 이 함수가 값을 반환하지 못할 수 있다는 것을 알 수 있음.
* 2.2. 타입 안정성: 억지로 값을 꺼내려 하면 std::bad_optional_access예외가 발생하여 실수로 유효하지 않은 값을 참조하는 것을 방지함.
* 2.3. 매직 넘버 제거됨: nullptr이나 -1이 에러인지 정상값인지 고민할 필요 없음.

## 3. 언제쓸까?


| 상황 | 방식 |
| --- | --- |
| 반환값이 없을 가능성이 논리적으로 존재할 때 | std::optional<T>|
|성능이 극도로 중요하거나 포인터로 충분할 때|T* 또는 const T* (혹은 iterator 구현)|
|데이터가 반드시 존재해야 하는 경우 | T& (참조 반환)|
|값 없음을 무조건 예외 상황으로 간주할 때|try-catch 또는 예외 던지기|

## 4. 느낀점:
구린 것 같다.
템플릿 해시맵을 만들 때 Get()에서 키가 존재하지 않을 때 반환을 뭘 해줄지 애매해 다음 코드처럼 Get을 구현했었다.
```cpp
	std::optional<T> Get(std::string key) const
	{
		int idx = GetHash(key);
		for (const Node& node : m_table)
		{
			if (node.Key == key)
				return node.Data;
		}
		return std::nullopt;
	}

```
그러나 래핑 구조만 봐도 패딩으로 손해를 너무 볼 것 같다. T객체 크기가 클수록 더 보게될거다. 예전엔 메모리를 손해보고 다른데서 이득을 취하는 방법을 많이 썼었는데 요새는 메모리가 아니라 캐시가 좁은게 문제라는걸 많이 느끼고 있어서 좋은 느낌이 안든다.

반환값이 복사되는것도 마음에 안든다. 물론 사용자입장에서 큰 객체라면 포인터를 담아서 크기가 작겠지만, 대부분 그 포인터로 가리키는 객체들이 메모리상에 연속되지 못할 것 같다.
