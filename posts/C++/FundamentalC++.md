# 클래스

## 생성자

### 상속관계 생성자소멸자 호출 순서
생성자: 자식생성자 호출 직후 부모생성자 블럭 실행. (자식->부모 지만 실행은 부->자)
소멸자: 자식소멸자 실행 이후 부모소멸자 블럭 실행. (자식->부모)
상식적인듯. 자식의 코드는 부모가 존재해야 가능하니까.

### 이니셜라이저 (초기화 리스트 initailization list)
초기화 리스트써서 더 빨라지는건 아님. 어셈 똑같이 나오는거 확인.

쓸 수 밖에 없는 상황이 있음. 이니셜라이저는 선처리 영역 제어가능함. 
즉, 부모 생성자 인자가 명시된 버전으로 호출 가능함
※이니셜라이저리스트의 순서로 초기화되지않음. 초기화 순서는 기존 선처리 순서를 따름.

### virtual 소멸자해야되는 이유
CParent* child = new CChild(); 처럼 다형성을 활용했을때 소멸자가 가상함수가 아니면 CParent의 소멸만 호출됨.
그렇다고 남발하면 vptr땜에 메모리 낭비되니까 ㄴㄴ

그런데 소멸자는 상속이 되지 않아서 virtual로 재정의된다는게 이상하다.
사실 객체의 소멸에는 객체를 해제하기 전에 정리하는 ‘소멸자’와 메모리를 해제하는 ‘파괴자’가 관여한다. 소멸자에 virtual을 붙이면 바로 이 파괴자를 가상 함수로 지정하는 것이다.
따라서 VTable에는 소멸자의 주소가아니라 파괴자의 주소가 들어간다.
단순히 말하면 소멸자는 이름이 달라서 오버라이딩이 안되니까 파괴자를 VTable에 등록하는거다.
[delete -> 파괴자 -> 소멸자] 순서로 진행된다.

스택에 객체를 생성하고 소멸시키면 ~Child가 바로 보이는데, 힙에다가 하면 scalar deleting destructor라는 함수를 호출하는데 그 안에서 delete가 다시 호출된다. 이게 파괴자이다.

그래서 VTable에 들어가는 파괴자가 같기 때문에 마치 소멸자가 일반 멤버 함수처럼 가상 함수로 동작할 수 있다.

### 암시적생성자
부모에게 생성자가있으면 자식은 부모생성자호출이 필요해져서 암시적생성자가생기더라
소멸자도 비슷함

### 복사생성자
자기를 참조자로 넣은 생성자임.
암시랑 명시때랑 선처리가 다름.
암시적 복사생성자 선처리: 부모 복사생성자 호출 -> 클래스 멤버 복사생성자 호출(얕복)
명시적: 부모 기본생성자 호출 ->클래스멤버 기본생성자 호출 -> 기본타입복사 -> 멤버가 배열인 경우 2, 3 중 골라서 실행 -> 멤버가 참조 타입이라면 대상 복사

복사라는게 그냥 다른 자리에 똑같은 값을 카피뜨는거라서 포인터 카피떴다가(얕은복사) 가리키던게 사라지면 곤란해짐. 맘에 안들면 명시적으로 복사 생성자를 작성하라는 의도.
그리고 CTest(const CTest& obj){ *this = obj; } //이거 무한루프임

### 복사 대입 연산자
CTest& operator = (const CTest& obj){}
처럼 생김.
 주의할건 
CTest t0;
CTest t2 = t0; // 이건 복사 생성자
CTest t3; t3 = t0; // 이건 복사 대입 연산자임.
복사 대입은 생성자가아니라 연산자임. 그래서 선처리 부분이 존재하지 않음.
암시적인 복사 대입 연산자는 부모 복사 대입연산자-> (멤버가 클래스면 복사 대입연산자, 기본타입이면 메모리복사,배열이면 타입따라)-> return *this; 순으로 진행됨.

## 상속
상속의 실체는 자식 클래스에서 부모의 멤버들을 특별히 정의하지않더라도 사용할 수 있다는 의미. 만약 사용할 수 없다면, 상속 불가로 인정할 수 있음.
1) 생성자는 상속 안됨. 호출만 함. 소멸자도 마찬가지.
2) 대입 연산자도 상속 안됨. 암시 복사 대입 연산자가 부모의 복사 대입 연산자를 호출하는거임.
3) 타입 변환 연산자는 상속 됨. 재정의된 일반 연산자도 상속 됨.


### 메모리구조
자식 공간엔 부모 메모리 구조가 그대로 포함돼있음. 
메모리 상엔 [ [ 부모구조 ] 자식구조 ] 이렇게 정렬됨.
부모랑 자식의 시작 주소가 같다고 볼 수 있음.

당연히 static은 전역으로 빠져있음.
static은 원래면 따로 밖에다 int CTest::s_val = 1; 처럼 정의 해줘야됨.
근데 난 걍 inline static으로 다 떼우긴 함 ㅋ

## 다중 상속
메모리는 [ [ 부모 1 ] [ 부모 2] 자식 ] 임.
부모 두번째부턴 자식이랑 주소가 다름.
```
CChild c;
int Ret1 = c.GetAVal();// A부모의 시작주소 thiscall
int Ret2 = c.GetBVal();// B부모의 시작주소 thiscall
```

## 다이아몬드 구조
최상위부모Top을 상속받은 중간 부모 A, B를 다중 상속받으면 Child에 Top의 멤버가 중복으로 상속됨. 컴파일러는 중복 중에 뭘 사용해야될지 모호하다고 경고줌. 사실 이건 A에 상속된거만 쓰겠다고 명시 (CMiddleA::m_top = 10;)해주면 되는데, 메모리 낭비가 생김.
-> 가상 상속이란게 제시됨

## 가상 상속
class A: virtual public Top{}
class B: virtual public Top{}
class Bottom : public A, public B{}
하면 중복 사라짐. Top의 생성자도 Bottom에서 한번만 호출됨.
메모리 구조는 [ [vbptr] [ A ] [ B ] [ Top ] 자식 ] 으로 보임 Visual C++에선.
vbptr은 가상 기저 테이블 포인터라고 (정식 명칭 아님) 부를건데. 가상 시저 클래스를 상속 받으면 이를 관리하기 위해 기본 포인터만큼 메모리를 확보하는 거임.
일단 vbptr생겨서 8바이트 생기기도하고, 클래스 멤버에 접근하려면 클래스 포인터의 타입 변환이 일어날 수 밖에 없는데, 이 때 가상 기저 테이블 포인터에 접근해야되서 성능 떨어짐;;

여러개를 가상 상속 받으면 이렇게 되기도 함. (CD2A랑 CD2B는 CD1을 가상 상속 받는 중)
class D3 : virtual public CD2A, virtual public CD2B
[[vbptr of CD3 ] [CD3멤버][CD1멤버][vbptr of CD2A][CD2A멤버] [vbptr of CD2B][CD2B멤버]

vfptr이랑 같이있으면 컴파일러따라 다른듯;;
일단 VC는 자식에서 선언된 가상함수가 없으면
[[vbtr][child][vfptr][parent]고
자식에서 새롭게 선언한 가상함수가 있으면 메인 vfptr이 생기면서
[[vfptr][vbptr][child][vfptr][parent]]로 나옴;; 이걸 어케 외우냐 안할란다.
몇몇 컴파일러는 이런 특정 부분은 구현하지 않는 경우도 있다더라.

## 클래스 포인터 형변환
자식 -> 부모는 다 되는데
Child* c = p; 같은 암시적 변환은 안됨.
static_cast, reinterpret, (Parent)등은 되더라.

※ 다중 상속에서는 형변환시에 포인터 값이 변할 수 있다. 2번째 부모로 변환하면 그 부모 시작 주소로 바뀌더라. reinterpret으로 형변환하면 주소값이 절대적으로 안바뀌는데, 그러면 잘못 접근해서 골치아플 수 있다.
※ 그러면 다중 상속이 아니어도 vbptr이나 vfptr이 껴있으면 오프셋따라 포인터 값이 변하겠다;;

### this

# 가상함수 (시험 단골이긴함)

다형성을 구현하기 위한 핵심 문법. 부모의 포인터를 사용하더라도 런타임 중에 객체를 판단하여 함수를 실행한다.

## 호출 구조
가상 함수는 함수랑 차이는 크게 없음. 그냥 코드 영역에 어딘가에 있고 컴파일러는 그 위치를 파악하고 있으며, call할 때 대입할 뿐.

일단 클래스 시작 위치에 가상 함수 테이블 포인터가 생성됨.
호출하면 vfptr->테이블에서가상함수찾아서->call 이더라.
```
pT->VFunc2();
mov eax, dword ptr [pT] // []는 참조값 읽기
mov edx, dword ptr[eax] // eax는 객체 시작주소 즉, vfptr을 edx로 옮김
mov eax, dword ptr[edx+4] // 테이블 두번째 요소 호출하려나봄. x86인가봄.
call eax
```

## 테이블은 언제 생길까
컴파일러가 컴파일 시점에 소스 코드에 정의된 모든 클래스에 대해서 가상 함수가 하나라도 있는 경우에 가상 함수 테이블을 생성한다. 그리고 각 클래스의 객체가 생성될 때 객체의 vfptr에 해당 클래스의 가상 함수 테이블 주소를 설정한다.

> 컴파일러가 가상 함수가 있는걸 확인하면 각 클래스 마다 가상 함수 테이블을 구성함.
> 컴파일러가 가상 함수 테이블을 생성할 때에는 각 항목에 들어갈 함수 정보와 인덱스를 묶어서 기억한다. (VFunc, 0)을 통해 vfptr의 0번째 인덱스를 호출하도록 어셈블리를 작성한다.

## vfptr을 어떻게 구분할까
자식의 생성자 선처리에선 부모의 생성자를 호출하는데, 이 때 부모의 테이블로 설정됐다가 이후 자식의 생성자가 진행되면 최종적으로 자식의 테이블로 설정된다.
만약 CParent* parent = (CParent*)child; 여도 parent가 나타내는 객체의 vfptr은 CChild의 가상 함수 테이블을 가리키고 있다.

## 가상함수는 포인터 참조로만 호출된다
```
Parent p1 = child; p1.VFunc(); // 부모꺼 호출
Parent& p2 = child; p2.VFunc(); // 자식꺼 호출
```

## 가상함수테이블 교체 타이밍
자식의 선처리는 아래 순서로 진행된다.
1) 부모의 생성자 호출
2) VTable교체
3) 멤버 생성자 호출

소멸자에서는 소멸자의 본체 블록에서 교체됨.
자식->부모 순으로.

## 생성자 소멸자에서 가상 함수를 호출하면?
위에 교체 타이밍을 적어놨는데, 가상 함수가 의도와는 다르게 vfptr따라 호출될 수 있다.
순수가상함수 호출해서 퓨어콜로 죽을 수도 있다;;

## 순수 가상 함수
가상 함수의 본체를 정의하지 않는 것. 상속 받는다면 반드시 구현해야 한다는 의도 전달.
정의 안되면 CRT의 _purecall을 바로 호출함. 이건 절 대 로 호출되면 안되는 함수임.

## 추상 클래스
순수 가상 함수를 하나라도 포함하면 추상 클래스라고 부름. 보통 인터페이스로 쓰려고 이렇게 만드는데, 얘네들은 생성하면 안되고 인터페이스 타입으로만 사용하겠다는 의도임.

## 다중 상속과 가상 함수
### vfptr개수
반드시 하나만 존재하진 않더라.
상속을 두 개받고 나도 처음 선언한 가상함수가 있는 경우엔 vfptr이 두 개 있었다.
내 가상함수는 첫 번째 vfptr에 있었다. 메모리 시작 위치라 그렇다.
가상 함수가 어떤  VTable에 포함되는지는 처음 선언된 클래스에 달렸다.
### 다중 상속일때 메모리 구조
컴파일러마다 다르긴 한데 vfptr을 가진 부모들이 상단을 차지하더라. 
생성자 호출 순서는 상속 순서 그대로다.
자세한건 p.685 참조

### this의 변화
cx레지스터에 담긴 this를 그냥 쓴다고 생각했는데, 멤버 함수에 접근할 때 멤버를 가진 클래스 오프셋 따라 변할수도있더라.
